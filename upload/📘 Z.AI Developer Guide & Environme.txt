Padrão Z.AI — “Especificação Absorvível” (como ele entende e vira código)
1) Como o Z.AI “absorve” informação

Ele performa melhor quando a informação vem em camadas, nessa ordem:

Identidade do produto (1 frase)
Nome + o que é + para quem + 1 diferencial.

Trilhas de restrição (o que NÃO pode falhar)
Stack obrigatório, página única vs multi, performance, offline, tipografia, layout fixo.

Contrato de UI/UX (o que fica igual sempre)
Cores, tipografia, grid, painéis, animações, dark mode.

Contrato de dados (onde salva, formato, chaves, IDs, versionamento)
IndexedDB/localStorage, schema de notas, árvore, histórico, busca.

Contrato de IA (API, modelos, streaming, erros, timeouts)
Config, endpoints, SSE, estados de loading, fallback, logs.

Features em “módulos” (com entradas/saídas)
Cada feature com: Trigger → Input → Process → Output → Persistência.

Critérios de pronto (Definition of Done)
“Roda local”, “sem telas quebradas”, “CRUD completo”, “cache ok”, “erro tratado”.

Entrega exigida
“Código completo executável + explicação de X/Y/Z”.

Se você mistura tudo, ele “perde prioridade”. Se você separa em contratos, ele implementa com menos divergência.
